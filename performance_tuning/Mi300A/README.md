### Mi300A (AMD/HIP)
## Systems: Viper, Hunter
# Notes: 
Please find the `disasm.sh` script in this directory, you can run it in the evolution build dir to disassemble the object files (pass the flag `GRACE_DUMP_GPU_RESOURCE_USAGE` to have them annotated). This will produce a few files, most notably `spillers.log` (example attached), that contains the (usually mangled) names of the top 20 kernels with the most register spills. In each line the first column is the number of bytes that spill into cache or main memory , the second column is the amount of registers per wavefront that the compiler assigned to this kernel. 
# What to do with this information
Generally speaking, low number of registers (~128) and low or no spilling is ideal. If the number of registers is low and the spilling is high it might be worth trying to pass `LoopBounds` to the policy ctor in Kokkos. The compiler might ignore it, in which case it figures that the spills are cold and it would rather keep the occupancy. If performance is bad in this case, rewrite the code or update the driver. If register allocation is max and still you see spilling (max is 512), you're in trouble. Might be time to split the kernel. If kernel splitting is not an option (e.g. all computations are interleaved), try aggressively scoping intermediates to let them fall out of scope and allow the compiler to reuse registers. Note that for this to be possible all functions have to be inlined in the kernel.
